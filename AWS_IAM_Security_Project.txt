<h3>Step 2: IAM Policy Creation ğŸ“œ</h3>

<p>Created a sophisticated JSON policy with granular permissions:</p>

<p><strong>Policy Objectives:</strong></p>
<ul>
<li>Allow all actions on EC2 instances tagged with <code>Env=development</code></li>
<li>Allow describe actions on all EC2 instances (read-only visibility)</li>
<li>Deny all delete and create actions on all EC2 instances (safety measure)</li>
</ul>

<p><strong>JSON Policy Structure:</strong></p>
<pre><code>{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "ec2:*",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "ec2:ResourceTag/Env": "development"
        }
      }
    },
    {
      "Effect": "Allow",
      "Action": "ec2:Describe*",
      "Resource": "*"
    },
    {
      "Effect": "Deny",
      "Action": [
        "ec2:Delete*",
        "ec2:Create*"
      ],
      "Resource": "*"
    }
  ]
}</code></pre>

<p><strong>Policy Components:</strong></p>
<ul>
<li><strong>Effect</strong>: Allow or Deny permissions</li>
<li><strong>Action</strong>: Specific API calls or operations</li>
<li><strong>Resource</strong>: Target AWS resources (ARNs or "*" for all)</li>
<li><strong>Condition</strong>: Additional constraints based on tags or other attributes</li>
</ul>

<h2>ğŸ” Security Features Implemented</h2>

<h3>1. Principle of Least Privilege</h3>
<ul>
<li>Users receive minimum necessary permissions</li>
<li>Environment-based access restrictions</li>
<li>Explicit deny rules for sensitive operations</li>
</ul>

<h3>2. Resource-Based Access Control</h3>
<ul>
<li>Tag-based policy conditions</li>
<li>Environment segregation (production vs development)</li>
<li>Granular action permissions</li>
</ul>

<h3>3. Defense in Depth</h3>
<ul>
<li>Multiple policy statements with different effects</li>
<li>Explicit deny rules override allow rules</li>
<li>Account-level and resource-level controls</li>
</ul>

<h2>ğŸš¨ Security Considerations</h2>

<h3>Policy Design Principles</h3>
<ol>
<li><strong>Explicit Deny Rules</strong> - Prevent accidental destructive actions</li>
<li><strong>Conditional Access</strong> - Tag-based resource targeting</li>
<li><strong>Read-Only Baseline</strong> - Default describe permissions for visibility</li>
<li><strong>Environment Isolation</strong> - Separate production and development access</li>
</ol>

<h3>Testing Methodology</h3>
<ol>
<li><strong>Positive Testing</strong> - Verify allowed actions work correctly</li>
<li><strong>Negative Testing</strong> - Confirm denied actions are blocked</li>
<li><strong>Boundary Testing</strong> - Test edge cases and policy limits</li>
</ol>
